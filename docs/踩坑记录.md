### 3.1 Docker 镜像拉取失败

**问题**：ustc/baidu/registry 镜像源均已失效  
**解决**：`daemon.json` 已更新为可用源：
```json
{
  "registry-mirrors": [
    "docker.1ms.run",
    "docker.xuynuan.me", 
    "docker.m.daocloud.io"
  ]
}
```

### 3.2 PostgreSQL 端口 5432 被 Windows Hyper-V 占用

**问题**：Windows Hyper-V 保留端口范围 4539-5557，导致 5432 无法绑定  
**解决**：改为端口 5600，同时修改了 `docker-compose.yml` 和 `application.yml`

### 3.3 Maven 依赖构建顺序

**问题**：单独运行 web 模块会报依赖找不到  
**解决**：必须从根目录先执行 `mvn clean install -DskipTests`，再启动 web 模块

### 3.4 Spring AI 1.0.0 GA 依赖名称变更

**问题**：GA 版本重命名了 starter artifact  
**解决**：
- `spring-ai-ollama-spring-boot-starter` → `spring-ai-starter-model-ollama`
- `spring-ai-pgvector-store-spring-boot-starter` → `spring-ai-starter-vector-store-pgvector`

### 3.5 pgvector0 依赖冲突

**问题**：Alibaba Maven 仓库找不到 `pgvector0` 依赖  
**解决**：从 dao/pom.xml 和父 POM 的 dependencyManagement 中移除 pgvector0，由 Spring AI starter 自动管理

### 3.6 循环依赖

**问题**：SpringAiConfig 创建 ChatClient → OllamaChatModel 自动发现 @Tool Bean → QuestionGenerateTool 注入 ChatClient → 形成循环  
**解决**：
1. SpringAiConfig 中 ChatClient 不注册任何 Tool（Tool 在 Service 层通过 `.tools()` 动态注册）
2. QuestionGenerateTool 改为注入 `OllamaChatModel`（而非 ChatClient），并使用 `@Lazy` 延迟注入
3. 循环链：`OllamaChatModel` → `ToolCallingManager` → `QuestionGenerateTool` →(lazy) `OllamaChatModel`

### 3.7 Mapper 扫描不到

**问题**：启动类在 `com.interview.agent.web`，Mapper 在 `com.interview.agent.dao.mapper`  
**解决**：在 `InterviewAgentApplication.java` 添加 `@MapperScan("com.interview.agent.dao.mapper")`

### 3.8 MyBatis-Plus 3.5.9 无 PaginationInnerInterceptor

**问题**：该类在 3.5.9 中不存在  
**解决**：简化 MyBatisPlusConfig，移除分页插件

### 3.9 前端 SSE 流式输出显示原始 `data:` 前缀

**问题**：前端使用 `fetch` + `ReadableStream` 直接读取 SSE 响应，未解析 SSE 协议格式，导致页面显示 `data:你好` 而非 `你好`  
**原因**：SSE 标准格式为 `data:内容\n\n`，前端直接拼接原始 chunk  
**解决**：添加 `parseSSEChunk()` 函数，逐行解析 SSE 数据并去掉 `data:` 前缀

### 3.10 后端历史消息重复

**问题**：`chatStream/chat` 方法先保存用户消息到数据库，`buildPrompt` 查询历史时查出该消息，又手动添加一次 `UserMessage`，导致当前用户消息在 prompt 中重复  
**解决**：调整执行顺序，先构建 Prompt，再保存用户消息

### 3.11 Qwen3.5 深度思考模式浪费大量时间

**问题**：Qwen3.5 默认启用深度思考（thinking），导致响应极慢，且思考过程输出到前端  
**原因**：Ollama 0.12+ 中思考型模型自动启用思考模式；`/no_think` 对 Qwen3.5 不生效  
**解决**：
1. 升级 Spring AI 从 1.0.0 到 1.1.0（1.0.x 不支持 `think` 参数）
2. 使用 `OllamaChatOptions.builder().disableThinking()` 默认关闭深度思考
3. 前端添加"深度思考"开关，根据模型是否支持来控制可用性
4. 支持深度思考的模型列表：qwen3、qwen3.5、deepseek-r1、deepseek-v3.1

### 3.12 Spring AI 1.0.0 不支持 Ollama `think` 参数

**问题**：Spring AI 1.0.0 的 `OllamaOptions` / `OllamaApi.ChatRequest` 均无 `think` 字段  
**解决**：升级到 Spring AI 1.1.0，使用 `OllamaChatOptions.builder().enableThinking()/.disableThinking()`

### 3.13 CORS 跨域请求被拦截

**问题**：前端（localhost:3311）请求后端（localhost:8180）被浏览器 CORS 策略拦截  
**解决**：在 `WebMvcConfig.java` 中配置 `allowedOrigins("http://localhost:5173", "http://localhost:3000")`；前端通过 vite proxy 代理 `/api` 到后端，开发模式下无需跨域

### 3.14 DeepSeek 模型 `<think\>` 标签泄露到前端

**问题**：DeepSeek 等支持深度思考的模型会在输出中包含 `<think\>...</think\>` 推理过程标签，直接显示给用户影响体验  
**解决**：前端 `renderMarkdown()` 函数中添加正则 `text.replace(/<think\>[\s\S]*?<\/think\>/g, '').trim()` 清理推理内容后再渲染

### 3.15 前端 Markdown 渲染和代码高亮

**问题**：AI 返回的 Markdown 格式内容（代码块、列表、标题等）在前端显示为纯文本  
**解决**：
1. 安装 `markdown-it` + `highlight.js` 依赖
2. 配置 markdown-it 的 highlight 回调使用 highlight.js 进行代码高亮
3. 引入 `highlight.js/styles/github.css` 样式
4. 添加 `word-wrap: break-word` 防止长文本溢出布局

### 3.16 chat_session 表缺少 title 字段

**问题**：对话会话表原本没有 `title` 字段，无法支持重命名功能，前端显示只能用 `domain`  
**解决**：
1. `schema.sql` 中 `chat_session` 表新增 `title VARCHAR(255)` 列
2. `ChatSession` Entity 和 `ChatSessionVO` 新增 `title` 字段
3. 对运行中的数据库执行 `ALTER TABLE chat_session ADD COLUMN IF NOT EXISTS title VARCHAR(255)`
4. 新增 `PUT /api/chat/sessions/{sessionId}/rename` 接口

### 3.17 流式输出时强制自动滚动，用户无法查看历史

**问题**：SSE 流式输出期间，每次收到新 chunk 都调用 `scrollToBottom()` 强制跳到底部，用户无法上滚查看之前的消息  
**解决**：
1. 添加 `isUserAtBottom` 状态，通过 `@scroll` 事件检测用户是否在底部附近（阈值 80px）
2. `scrollToBottom(force)` 仅在 `isUserAtBottom=true` 或 `force=true` 时自动滚动
3. 用户主动上滚时不打断，滚回底部后恢复自动跟随
4. 用户发送新消息时，强制 `isUserAtBottom = true` + `scrollToBottom(true)`

### 3.18 流式输出时 Markdown 代码高亮不生效

**问题**：SSE 流式输出过程中，Markdown 渲染为纯文本，代码块没有语法高亮，只有刷新页面后才显示正确  
**原因**：流式输出时代码块可能未关闭（如 ` ```java ` 还没有结束的 ` ``` `），`markdown-it` 无法识别为代码块，`hljs.highlight` 不触发  
**解决**：
1. 每条消息新增 `streaming` 标记，流式输出时为 `true`，结束时为 `false`
2. `renderMarkdown(text, streaming)` 在流式模式下统计 ` ``` ` 出现次数，奇数则自动追加 ` \n``` ` 补全未关闭的代码块
3. 这样代码块始终能被 `markdown-it` 正确解析并触发 `hljs.highlight` 实时高亮

### 3.19 侧边栏操作按钮太小难以点击

**问题**：对话历史的编辑和删除使用 14px 的 `el-icon`，太小且无文字说明，用户难以发现和点击  
**解决**：改用 `el-button`（text 类型）同时显示图标 + 文字（"重命名"、"删除"），增大可点击区域和视觉辨识度

### 3.20 删除/重命名会话返回 403 错误

**问题**：前端调用 `DELETE /api/chat/sessions/{id}` 和 `PUT /api/chat/sessions/{id}/rename` 返回 403  
**原因**：`application.yml` 中配置了 MyBatis-Plus 全局逻辑删除（`logic-delete-field: status`），导致所有 `mapper.delete()` 被转为 `UPDATE SET status='DELETED'`，但 `chat_session` 和 `chat_message` 表没有 `status` 字段，SQL 执行失败  
**解决**：
1. 移除 `application.yml` 中的全局逻辑删除配置
2. 在 `KnowledgeDocument` 实体的 `status` 字段上单独添加 `@TableLogic(value = "ACTIVE", delval = "DELETED")` 注解
3. 这样只有 `knowledge_document` 表使用逻辑删除，其他表正常物理删除

### 3.21 CORS 403：前端端口未加入白名单

**问题**：删除/重命名会话仍然返回 403 `Invalid CORS request`  
**原因**：前端运行在 `localhost:3311`，但 `WebMvcConfig.java` 的 `allowedOrigins` 只有 `5173` 和 `3000`；DELETE/PUT 等非简单请求触发 OPTIONS 预检，预检被 CORS 拦截  
**解决**：在 `allowedOrigins` 中添加 `"http://localhost:3311"`

### 3.22 流式输出 Markdown 样式不实时渲染（Vue 响应式问题）

**问题**：SSE 流式输出时，AI 回复显示为原始 Markdown 文本（如 `### 标题`、`- 列表`），刷新页面后才正确渲染为 HTML 样式  
**原因**：之前使用 `v-html="renderMarkdown(msg.content, msg.streaming)"` 在模板中调用函数，Vue 的响应式系统对深层属性变化（`msg.content`）的追踪不可靠，导致 `v-html` 不更新  
**解决**：
1. 消息对象新增 `renderedHtml` 属性，存储渲染后的 HTML
2. `v-html` 直接绑定 `msg.renderedHtml`，不再调用函数
3. 每次内容更新时，通过 `updateMessageContent()` 函数**替换整个数组元素**（`messages.value[msgIndex] = { ...msg, content, streaming, renderedHtml }`），确保 Vue 检测到引用变化
4. `renderToHtml()` 在 `streaming=true` 时自动补全未关闭的代码块

### 3.23 流式输出 Markdown 样式仍不渲染（Vue 虚拟 DOM diff 问题）

**问题**：即使改用 `msg.renderedHtml` 属性 + 替换数组元素的方式，SSE 高频更新时 Vue 的虚拟 DOM diff 仍然可能跳过 `v-html` 的实际 DOM 更新，导致 Markdown 始终显示为原始文本  
**解决**：
1. 新增 `messageRefs` 对象，存储每个消息 DOM 元素的引用
2. `updateMessageContent()` 中除了更新响应式数据外，还通过 `nextTick` **直接操作 DOM innerHTML**
3. 比对 `el.innerHTML !== renderedHtml` 后才赋值，避免不必要的 DOM 操作
4. 双保险：响应式 + 直接 DOM 操作，确保无论如何都能正确刷新

### 3.24 SSE 流首条消息携带 sessionId JSON 污染内容

**问题**：后端每次流式请求（不仅是首次）都在流的第一条发送 `{"sessionId":"xxx"}`，前端只在第一次对话时过滤了它；后续对话中该 JSON 字符串混入了 AI 回复内容并直接显示出来  
**原因**：`ChatController.chatStream()` 使用 `Flux.concat(sessionIdEvent, contentStream)` 将 sessionId 作为第一个事件发出，但前端之前仅在 `!currentSessionId.value` 时才提取和移除它  
**解决**：
1. 新增 `extractSessionId(buffer)` 函数，**始终从 buffer 中检测并移除** sessionId JSON 行（不管是否已有会话ID）
2. 改进 `parseSSEChunk()` 函数，在解析时也过滤掉 `{...sessionId...}` 格式的数据
3. 双重过滤确保 sessionId 不会混入 AI 回复内容

### 3.25 SSE 流式输出 Markdown 标题不渲染（换行符丢失 — 根因修复）

**问题**：SSE 流式输出时，`**加粗**` 能渲染但 `### 标题` 不能，显示为原始文本  
**根因**：`parseSSEChunk()` 函数用 `result += data` 直接拼接每条 `data:` 行的内容，**丢失了行间换行符**

后端 Spring WebFlux `Flux<String>` 输出格式：
```
data:你好我是助手。
data:
data:### 面试题解析
data:
data:**参考答案**
```

错误拼接为：`你好我是助手。### 面试题解析**参考答案**`
→ `###` 不在行首 → markdown-it 不识别为标题 ❌
→ `**` 在段落中间 → 恰好能被 inline 解析器识别 ✅

**参考主流方案（ChatGPT / Streamdown / streaming-markdown）的核心做法**：
1. **保留原始文本结构**：每个 data: 行用 `\n` 连接，不丢失换行
2. **增量 DOM 渲染**：直接操作 innerHTML + Vue 响应式双保险
3. **容忍不完整 Markdown**：流式时自动补全未关闭的代码块

**修复**：
1. 重写 `parseSSEChunk()` → 改名为 `parseSSEDataLines()`，使用 `parts.join('\n')` 保留换行
2. 清理冗余逻辑，代码结构更清晰
3. 参考了 [streaming-markdown](https://github.com/thetarnav/streaming-markdown) 和 [Streamdown](https://juejin.cn/post/7568796644347723776) 的实现思路

### 3.26 SSE 流式输出文字内容丢失（buffer 清空导致截断）— 最终根因修复

**问题**：流式输出时不仅 Markdown 样式不渲染，**连文字内容本身都在丢失**
- 流式输出时显示：`封装数据捆绑，暴露必要，。限制对象的直接数据完整性，降低，代码修改`
- 刷新后完整：`封装是将数据与方法捆绑，仅暴露必要接口，隐藏内部实现细节。它通过限制外部对对象的直接访问，保护数据完整性，降低耦合度，便于代码维护和修改。`

**根因**：`readSSEStream()` 中每次解析后直接 `buffer = ''` 清空整个缓冲区！
```
while 循环中：
  buffer += decoder.decode(value, { stream: true })  // 追加新数据
  const parsed = parseSSEDataLines(buffer)             // 提取 data: 内容
  assistantContent += parsed                           // 累加到结果
  buffer = ''  // ← 致命错误！清空了整个 buffer，丢弃未完成的数据行！
```
网络传输中一个 TCP 包可能切断在任意位置（比如 `data:封装是` 被切成两包），`buffer = ''` 直接把后半部分丢掉了。

**参考主流方案（CSDN / OpenAI SDK / Vercel AI SDK 的标准模式）重写**：
1. **`extractSSEDatas(buffer)`** — 只从 buffer 中提取已完成的 `data:` 行；如果 buffer 不以 `\n` 结尾，说明最后一行未完成，保留在 `remainingBuffer`
2. **TextDecoder `stream:true`** — 正确处理被切断的多字节 UTF-8 字符（中文 3 字节）
3. **流结束时 `decoder.decode(new Uint8Array(0), { stream: false })`** — 刷新 TextDecoder 内部缓冲区
4. **空行保留为换行符** — 确保 `###` 在行首、段落之间有正确间隔

### 3.31 bge-m3 Embedding 全部返回 NaN（HTTP 500）

**问题**：Ollama bge-m3 对所有文档做 Embedding 都报 `HTTP 500 - {"error":"failed to encode response: json: unsupported value: NaN"}`  
**根因**（GitHub Issue [#13572](https://github.com/ollama/ollama/issues/13572)）：llama.cpp 的 **Flash Attention 对 BERT 类模型有 F32→F16 精度溢出 bug**。当 K/V 张量为 F32 时被强制转换为 F16，F16 最大值约 65504，中间值超过此限制溢出为 Inf → softmax 传播后变 NaN → JSON 无法序列化  
**解决**：设置环境变量 `OLLAMA_FLASH_ATTENTION=false`（用户级），重启 Ollama 服务即可  
**注意**：
- 设 `0` 无效，必须用 `false`
- 所有基于 BERT 架构的嵌入模型（bge-m3、intfloat-multilingual-e5-large 等）均受影响
- 替代方案：换用非 BERT 架构的嵌入模型如 `nomic-embed-text`、`mxbai-embed-large`

### 3.32 向量存储日志统计 bug — 成功数与总数混淆

2. 全部失败（`successCount == 0`）时抛异常，不写入数据库
3. `knowledge_document.chunk_count` 记录实际成功入库数
4. 日志明确输出"成功 X/Y 个分块"

### 3.33 Ollama GPU 不加速 — CPU 100% 而 GPU 空闲

**问题**：Ollama 运行 `qwen3.5:4b` 模型时 CPU 占用 100%，GPU 占用不到 10%，推理速度仅 1.6 tokens/s，导致智能对话几乎不可用（请求超时或极慢）。而 `qwen3:1.7b` 模型却能正常使用 GPU  
**根因**：Ollama 0.21.0 新引擎对 `qwen3.5:4b` 的 compute graph（计算图）占用 **4.3 GiB** 显存，RTX 2060 仅 6GB 总显存，计算图吃掉大部分后只剩 ~700MB 给模型权重，导致 **32/33 层被迫跑在 CPU 上**（仅 1 层在 GPU）  
**解决**：
1. **开启 Flash Attention**：设置环境变量 `OLLAMA_FLASH_ATTENTION=1`，计算图从 4.3 GiB → **727 MiB**（降 83%）
2. **降低上下文长度**：设置 `OLLAMA_CONTEXT_LENGTH=4096`（从默认 8192 降低），进一步减少 KV 缓存占用
3. **Spring AI 配置同步**：`application.yml` 中添加 `num-ctx: 4096`，确保请求参数与服务端一致
4. **创建启动脚本**：`start_ollama.bat` 在启动 Ollama 前设置环境变量（Ollama 系统托盘启动不会读取 setx 设置的变量）
5. **关闭占用显存的程序**：如 MuMu 模拟器等

**优化效果**：

| 指标 | 优化前 | 优化后 |
|------|--------|--------|
| GPU 层数 | 2/33 (6%) | 32/33 (97%) |
| 推理速度 | 1.6 tokens/s | 21.7 tokens/s (13.5倍) |
| 模型启动时间 | 58秒 | 4.4秒 |
| 计算图占用(CUDA) | 4.3 GiB | 727 MiB |

**注意事项**：
- `OLLAMA_FLASH_ATTENTION` 对 BERT 类嵌入模型（如 bge-m3）有精度溢出 bug（见 3.31），但对话模型不受影响
- `OLLAMA_CONTEXT_LENGTH` 是 Ollama 服务端默认值，Spring AI 请求中的 `num-ctx` 参数优先级更高，会覆盖服务端默认值
- 环境变量通过 `setx` 永久设置，但 Ollama 系统托盘应用启动时不会读取，必须通过 `start_ollama.bat` 或在 PowerShell 中手动设置后启动

### 3.37 PromptTemplate 读取 .st 模板文件中文乱码（根因修复）

**问题**：3.36 的 `.mvn/jvm.config` 方案仅对 Maven CLI 启动有效，IDE 启动时 `.st` 模板文件内容仍然乱码（如 `浣犳槸涓�浣嶄笓涓氱殑闈㈣瘯鏅鸿兘鍔╂墜`）  
**根因**：`PromptTemplate(Resource)` 构造函数内部使用 `Charset.defaultCharset()` 读取文件内容。Windows 中文环境下默认编码为 GBK，而 `.st` 文件是 UTF-8 编码，导致 UTF-8 字节被当作 GBK 解码产生乱码  
**解决**：不使用 `PromptTemplate(Resource)` 构造函数，改为手动用 `StandardCharsets.UTF_8` 显式读取文件内容，再用 `PromptTemplate(String)` 构造：
```java
// 修改前（依赖 JVM 默认编码，Windows 下为 GBK）
PromptTemplate template = new PromptTemplate(systemPromptResource);

// 修改后（显式 UTF-8，不依赖 JVM 编码设置）
String templateContent;
try {
    templateContent = systemPromptResource.getContentAsString(StandardCharsets.UTF_8);
} catch (Exception e) {
    log.error("[Prompt] 读取系统模板文件失败", e);
    throw new BusinessException(ResultCode.CHAT_ERROR);
}
PromptTemplate template = new PromptTemplate(templateContent);
```
**影响文件**：
- `ChatServiceImpl.java` — `interview-system.st` 模板读取
- `RagServiceImpl.java` — `rag-enhanced-answer.st` 模板读取

**优点**：无论从 IDE、Maven CLI 还是任何方式启动，都能正确读取中文模板内容

### 3.38 Resource.getContentAsString() 受检异常未处理

**问题**：3.37 中使用 `systemPromptResource.getContentAsString(StandardCharsets.UTF_8)` 替代 `PromptTemplate(Resource)`，但 `Resource.getContentAsString(Charset)` 声明了 `throws IOException`（受检异常），直接调用导致编译错误：`Unhandled exception type IOException`

```java
    log.error("[Prompt] 读取系统模板文件失败", e);

```
**影响文件**：

- `RagServiceImpl.java` — `buildRagPrompt()` 方法

### 3.39 Spring AI VectorStore.delete() 类型不匹配（varchar vs uuid）

**问题**：调用 `vectorStore.delete(List.of("a252503a-03be-49cc-bf7d-3e87177bbdfc"))` 删除向量时报错 `operator does not exist: character varying = uuid`  
**原因**：Spring AI PgVectorStore 的 `vector_store` 表的 `id` 列类型为 `varchar`，但 `delete()` 方法生成的 SQL 将参数作为 `uuid` 类型传入（`'xxx'::uuid`），PostgreSQL 严格类型检查导致不匹配  
**解决**：放弃使用 `vectorStore.delete()`，改用 `JdbcTemplate` 直接执行 SQL：
```java
private void deleteVectorsByDocId(Long docId) {
    int deleted = jdbcTemplate.update(
        "DELETE FROM vector_store WHERE metadata->>'doc_id' = ?",
        String.valueOf(docId)
    );
    log.info("PGVector 向量数据已清理: docId={}, 删除向量数={}", docId, deleted);
}
```
**优势**：通过 `metadata->>'doc_id'` 精确匹配文档 ID，无需知道向量 ID，且避免类型转换问题  
**影响文件**：
- `RagServiceImpl.java` — `deleteVectorsByDocId()` 方法

### 3.40 @TableLogic 导致 updateById 无法更新 status 字段

**问题**：文档删除时调用 `doc.setStatus("DELETED")` + `documentMapper.updateById(doc)`，日志显示"文档已标记删除"但数据库中 status 仍为 ACTIVE  
**原因**：`KnowledgeDocument` 实体的 `status` 字段标注了 `@TableLogic(value = "ACTIVE", delval = "DELETED")`，MyBatis-Plus 对逻辑删除字段有特殊处理：
1. `updateById` 生成的 SQL 会自动添加 `WHERE status = 'ACTIVE'` 条件（正确）
2. 但 **SET 子句中不会包含 status 字段**，因为 MyBatis-Plus 认为逻辑删除应通过 `deleteById` 触发，而非手动 set
3. 所以 `SET status='DELETED'` 被忽略了，实际 SQL 变成 `UPDATE ... SET updated_at=? WHERE id=? AND status='ACTIVE'`

**解决**：改用 `documentMapper.deleteById(documentId)`，让 MyBatis-Plus 自动将 status 设为 DELETED：
```java
documentMapper.deleteById(documentId);
```
**注意**：`deleteById` 不会自动更新 `updated_at` 字段，如需记录删除时间需额外处理  
**影响文件**：
- `RagServiceImpl.java` — `deleteDocument()` 方法

### 3.41 RAG 检索返回 0 条文档 — domain 元数据大小写不匹配

**问题**：前端开启"知识库增强"后，RAG 向量检索始终返回 0 条文档，日志如下：
```
[RAG-Search] 向量检索: query='什么是多态', domain='Java', topK=5, threshold=0.5
[RAG-Search] 检索完成: 命中 0 条文档
```

**根因**：前端 ChatView 的领域选择器 value 使用大写（`"Java"`、`"Python"`），而知识库上传页面 KnowledgeView 的 value 使用小写（`"java"`、`"python"`）。数据库 vector_store 中 253 条向量的 domain 元数据全部为小写 `'java'`，但检索时 filter 表达式为 `domain == 'Java'`，大小写不匹配导致过滤后 0 条结果。

**验证**：

```sql
SELECT COUNT(*) FROM vector_store WHERE metadata->>'domain' = 'java';   -- 253 条
```

**面试亮点**：RAG 系统中元数据过滤是精确匹配（exact match），大小写敏感。前后端数据约定不一致是隐蔽 bug 的常见来源，最佳实践是在写入时统一归一化（normalize），检索时也做防御性转换。

### 3.43 MyBatis StdOutImpl 导致日志混乱重复输出

**问题**：终端日志非常混乱，MyBatis 的 SQL 日志和 Spring 业务日志混在一起，且聊天历史查询把所有历史消息完整内容都 dump 到控制台，导致一次 RAG 请求的日志输出长达数百行，极难排查问题

2. Spring AI 的 `ollama`、`chat.client`、`vectorstore` 三个包都设为 DEBUG 级别，产生大量冗余的框架内部日志
3. `RagServiceImpl` 的 DEBUG 日志输出了检索结果的完整文档内容和 System Prompt 全文，进一步膨胀日志

**解决**：
1. **移除 `log-impl: org.apache.ibatis.logging.stdout.StdOutImpl`**：让 MyBatis SQL 日志走 logback 通道，由 logback 统一控制格式和级别
2. **Spring AI 日志 DEBUG → WARN**：`application.yml` 和 `logback-spring.xml` 同步调整，只输出警告和错误
3. **移除 RagServiceImpl 冗余 DEBUG 日志**：删除 `结果[N] 完整内容`、`RAG上下文内容`、`System Prompt内容` 三条 DEBUG 输出，INFO 级别的摘要信息已足够排查

**面试亮点**：`StdOutImpl` 是 MyBatis 的"简易日志"，直接写 stdout，不经过任何日志框架。生产环境应使用 `Slf4jImpl` 或 `Log4j2Impl`，让所有日志统一走日志框架，才能做到格式统一、级别可控、按需输出

**影响文件**：
- `application.yml` — 移除 `log-impl` 配置 + Spring AI 日志级别调整
- `logback-spring.xml` — Spring AI logger 级别 DEBUG → WARN
- `RagServiceImpl.java` — 移除 3 条冗余 DEBUG 日志

### 3.44 RAG 提示词引用格式 — LLM 输出占位符而非实际来源

**问题**：RAG 增强回答中，参考资料部分输出了字面量占位符而非实际来源信息：
```
### 参考资料
1. 面向对象编程基础（领域/文档标题/段落）：
   "封装是面向对象编程中的核心概念..."
   - 来源：《面向对象编程基础》（领域/文档标题/段落）
```
其中"面向对象编程基础"是 LLM 编造的标题，"领域/文档标题/段落"是原封不动输出的占位符

**根因**：提示词模板 `rag-enhanced-answer.st` 第3条写的是：
```
3. 引用参考资料时，标注来源信息（领域/文档标题/段落）
```
括号中的"领域/文档标题/段落"被 LLM 当成了格式模板字面量，而不是元数据字段的说明。同时，上下文注入的格式是 `【来源: java / Java基础面试题-小林coding - 第28段】`，但提示词没有告诉 LLM 使用这个已有格式

**解决**：将提示词第3条改为：
```
3. 引用参考资料时，使用上下文中【来源: 领域 / 文档标题 - 第N段】的格式标注出处，不要编造来源
```
关键改动：
1. 明确引用上下文中已有的 `【来源: ...】` 格式，而非让 LLM 自行构造
2. 添加"不要编造来源"约束，防止 LLM 凭空捏造标题

**面试亮点**：Prompt Engineering 中，"示例驱动"远比"描述驱动"有效。与其告诉 LLM "标注来源信息（领域/文档标题/段落）"这种抽象描述，不如直接给出具体的格式示例 `【来源: java / Java基础面试题-小林coding - 第28段】`，让 LLM 一看就知道该怎么输出

**影响文件**：
- `rag-enhanced-answer.st` — 第3条提示词改写

### 3.45 RAG 向量数据质量差 — PDF 文本噪声导致检索和生成效果下降

**问题**：RAG 检索到的文档片段包含大量格式噪声（页眉页脚、空行、PDF 断行），导致：
1. 向量质量差 — 噪声文本的 Embedding 语义模糊，检索精度低
2. Prompt 污染 — 检索到的噪声片段注入 LLM 上下文，降低回答质量
3. Token 浪费 — 空行和无意义内容占用宝贵的上下文窗口

**根因**：
1. `PagePdfDocumentReader`（PdfBox）按页提取，保留了页眉页脚、日期戳、大量空行
2. PDF 文本提取时中文句子被拆成多行（PDF 断行问题），如"多态是指子类可以替换父类，在实际的代码运行\n过程中，调用子类的方法实现"
3. 无文本清洗步骤 — 提取后直接进入分块，噪声直接进入向量数据库
4. `TokenTextSplitter` 只按 Token 数切割，不支持重叠，容易切断语义边界

**解决**：三层优化

**① DocumentTextCleaner（自定义 DocumentTransformer）**
- 规则化去噪：正则匹配页眉（日期戳）、页码、重复标题行
- 空白符规范化：多空格→单空格，多空行→单空行
- PDF 断行修复：中文行尾无标点时合并为连续段落
- 控制字符清理：移除 0x00-0x1F, 0x7F-0x9F
- 短行过滤：移除过短的无意义行（如独立页码）

**② SmartTextSplitter（自定义 DocumentTransformer）**
- 段落优先分割：优先在双换行符（段落边界）处分割，保持语义完整
- 重叠窗口：相邻分块保留 12.5% 重叠（业界推荐 10%~20%），解决边界断裂问题
- 中文感知：在中文句末标点处优先分割
- 长文本递归分割：超长段落递归切分到句子级别

**③ PDF 解析优化**
- 优先使用 `ParagraphPdfDocumentReader`（利用 PDF 目录信息按段落提取）
- 降级为 `PagePdfDocumentReader`（按页提取，兼容无目录的 PDF）

**优化前后对比**：
```
优化前（直接 PagePdfDocumentReader → TokenTextSplitter）：
  "2026/3/3  00:25                                            Java基础面试题     |  小林coding |  Java面试学习\n\n\n          多态解决了什么问题？\n\n          多态是指子类可以替换父类，在实际的代码运行过程中，调用子类的方法实现。多态这种特性也\n          需要编程语言提供特殊的语法机制来实现，比如继承、接口类。"

优化后（ParagraphPdf → DocumentTextCleaner → SmartTextSplitter）：
  "多态解决了什么问题？\n多态是指子类可以替换父类，在实际的代码运行过程中，调用子类的方法实现。多态这种特性也需要编程语言提供特殊的语法机制来实现，比如继承、接口类。"
```

**面试亮点**：RAG 系统中"数据质量决定上限"——再好的检索算法和 LLM，如果输入数据是垃圾，输出也是垃圾（Garbage In, Garbage Out）。文本清洗是 RAG 最被低估但 ROI 最高的优化环节。

**影响文件**：
- `DocumentTextCleaner.java` — 新增自定义文本清洗 DocumentTransformer
- `SmartTextSplitter.java` — 新增语义感知分块器
- `RagServiceImpl.java` — 导入流程集成清洗+分块优化
- `SpringAiConfig.java` — 注册新 Bean

### 3.45 DocumentTextCleaner 短行过滤误删代码标点行

**问题**：`DocumentTextCleaner.isShortNoiseLine()` 会将长度 < `minLineLength`（默认4）的行视为噪声并删除，但 Java 代码中的 `}`、`};`、`);` 等代码标点行长度为1-2，被误删导致代码结构破坏

**原因**：短行过滤逻辑只考虑了中文文本场景，没有考虑代码块中的短标点行

**解决**：
1. 新增 `CODE_PUNCTUATION_LINE` 正则：`^[};\\)\\]]+[;,.]?$`，匹配 `}`、`};`、`);`、`]);` 等代码标点行
2. `isShortNoiseLine()` 中优先判断：如果匹配 `CODE_PUNCTUATION_LINE`，直接返回 `false`（不是噪声）
3. 同时优化：包含中文的短行（`CONTAINS_CHINESE`）和以中文句末标点结尾的短行（`CHINESE_SENTENCE_END`）也不视为噪声

**影响文件**：`DocumentTextCleaner.java`

### 3.46 向量存储速度慢 — Ollama 本地 Embedding 推理瓶颈

**问题**：上传一份 PDF 文档（131条分块）存入向量数据库耗时约25秒（13:29:09 → 13:29:34），用户怀疑是否正常

**原因**：瓶颈在 Ollama 本地 Embedding 推理（bge-m3 模型），每条约 190ms，131 × 190ms ≈ 25秒。PGVector 数据库写入本身很快

**结论**：**速度正常**，这是本地 Ollama Embedding 的典型性能

**优化方向**（如需加速）：
1. **批量 Embedding**：Ollama 支持 `/api/embeddings` 批量请求，减少 HTTP 往返开销
2. **云端 Embedding API**：使用 OpenAI / 智谱 / 阿里云等云端 Embedding 服务，延迟更低
3. **异步导入**：文档导入改为异步任务，前端轮询进度，不阻塞用户操作
4. **GPU 加速**：确保 Ollama 使用 GPU 推理（参考踩坑记录 3.33）

### 3.47 PDF 代码块逐行拆散 — PDF 文本提取固有局限

**问题**：PGVector 数据库中抽查发现，PDF 中的 Java 代码块被逐行拆散存储，每行代码独立成行，中间有空行，且代码语言标记 `java` 独占一行

**示例**：
```
count++;
    +
 }
    +

    +
 public synchronized int getCount() {
    +
 return count;
```

**原因**：`PagePdfDocumentReader`（Apache PdfBox）按页提取 PDF 文本时，代码块中的每一行都是独立的文本元素，无法区分"代码块"和"正文段落"。`DocumentTextCleaner.fixPdfLineBreaks()` 只合并中文行（行尾无标点时合并），不合并代码行

**影响**：
1. 代码块语义不完整 — 一个方法被拆成多个 chunk
2. 部分短文本块仅 94 字符，信息量不足
3. 代码语言标记 `java` 独占一行成为噪声

**解决方案**（待实施）：
1. **Markdown/HTML 格式文档优先**：如果源文档有 Markdown 版本，优先使用 TextReader 而非 PDF 解析
2. **代码块检测与合并**：在 DocumentTextCleaner 中增加代码块检测逻辑（连续缩进行 + 代码标点行模式），将代码块合并为一个整体
3. **结构化 PDF 解析**：使用更高级的 PDF 解析库（如 pdfplumber / Camelot / Unstructured.io）识别代码块区域
4. **接受现状**：对于面试知识库，纯文本段落（概念解释、对比分析）的质量已经很好，这些才是 RAG 检索最核心的内容；代码块的碎片化对检索影响相对有限

### 3.48 Ollama 不支持 Reranker 模型的 /v1/rerank 端点

**问题**：原计划用 Ollama 部署 bge-reranker-v2-m3 做 Reranking，但发现 Ollama 没有 `/v1/rerank` 端点

**原因**：
- Ollama 截至 2026-04 仍未实现 Reranker 模型类型的原生支持（[GitHub Issue #10467](https://github.com/ollama/ollama/issues/10467)）
- 虽然 `ollama pull bona/bge-reranker-v2-m3` 可以下载模型，但只能通过 chat API 做"提示词打分"
- 通过 chat API 让 LLM 打分本质是 **LLM Rerank**，不是 **Cross-Encoder Rerank**
- 用 1.7b 小模型做 LLM Rerank 效果差：输出不稳定、格式不可控、延迟高

**LLM Rerank vs Cross-Encoder Rerank 的本质区别**（面试高频）：

| 维度 | Cross-Encoder Rerank | LLM Rerank |
|------|---------------------|------------|
| 架构 | BERT 分类头 → 直接输出分数 | 自回归生成 → 输出文本 |
| 训练目标 | 专门训练 (query, doc) 相关性 | 训练文本续写/对话 |
| 输出 | 精确浮点分数（如 0.87） | 文本（如"8分"），需解析 |
| 准确性 | 高（专门优化） | 低（小模型尤其差） |
| 延迟 | 50-150ms（批量推理） | 数秒（多次 LLM 调用） |

**解决**：改用 HuggingFace TEI (Text Embeddings Inference) Docker 容器部署 Reranker

**TEI 优势**：
1. HuggingFace 官方推理服务器，Rust 实现，生产级稳定
2. 原生支持 `/rerank` 端点：`POST /rerank`，输入 query + texts，输出排序结果 + 分数
3. Docker 一键部署，和现有 PostgreSQL 容器架构一致
4. CPU 模式零 VRAM 占用，GPU 模式 ~2.5GB VRAM
5. bge-reranker-v2-m3 中文表现优秀，568M 参数

**影响文件**：
- `docker-compose.yml` — 新增 tei-reranker 容器
- `application.yml` — 新增 app.reranker.* 配置
- `RerankingDocumentPostProcessor.java` — 核心精排逻辑
- `RerankerProperties.java` — 配置属性映射
- `TeiRerankerApi.java` — TEI API 数据结构
- `RagServiceImpl.java` — 集成 Reranker（扩大召回 + 精排）
- `KnowledgeSearchTool.java` — Tool Calling 模式集成 Reranker

### 3.49 Reranker 降级策略 — 服务不可用时保证系统可用性

**问题**：Reranker 是外部服务（TEI Docker 容器），可能因网络/启动/资源等原因不可用

**解决**：在 `RerankingDocumentPostProcessor.rerank()` 中实现优雅降级：
1. `app.reranker.enabled=false` 时跳过精排，走原有纯向量检索逻辑
2. TEI 服务不可用时（连接失败/超时），自动降级为原始检索结果
3. JSON 序列化失败时，降级为原始检索结果
4. 所有降级场景都有 WARN 日志记录，便于排查

**设计原则**：Reranker 是"增强型"组件，不是"必需型"组件。系统核心功能（对话+检索）不应依赖 Reranker 的可用性

### 3.50 RAG 两阶段检索架构 — 向量检索扩大召回量

**问题**：集成 Reranker 后，向量检索的 Top-K 需要从 5 扩大到 20，否则 Reranker 没有足够的候选集

**原因**：
- 原架构：向量检索 Top-5 → 直接拼入 Prompt（单阶段）
- 新架构：向量检索 Top-20 → Reranker 精排 Top-5 → 拼入 Prompt（两阶段）
- 如果只检索 Top-5 再 Rerank，等于没有召回扩大，精排的意义大打折扣

**解决**：
1. `RerankerProperties.candidateCount=20` — 向量检索阶段召回 20 个候选文档
2. `RerankerProperties.topK=5` — Reranker 精排后保留 5 个最相关文档
3. `RagServiceImpl.searchSimilarDocs()` 根据 `rerankerEnabled` 动态调整 topK：
   - Reranker 启用：topK=20（扩大召回）
   - Reranker 关闭：topK=5（原有逻辑）
4. `KnowledgeSearchTool` 同样逻辑

**面试知识点**：两阶段检索是 RAG 生产系统的标配
- 第一阶段（召回）：向量检索 Top-K（快但粗，Embedding 近似匹配）
- 第二阶段（精排）：Cross-Encoder 对 Top-K 逐一打分 → 取 Top-N（慢但准）
- 候选集太小可能漏掉相关文档；候选集太大 Rerank 延迟线性增加
- 业界推荐：召回 Top-20~50，精排 Top-5~10

### 3.53 TEI Docker 容器 OOM — bge-reranker-v2-m3 CPU 模式内存不足

**问题**：TEI 容器启动后，模型加载到 "Warming up model" 阶段就被 OOM Kill，反复重启

**原因**：
1. Docker Desktop 默认只分配 **7.675GB** 内存
2. bge-reranker-v2-m3（568M 参数）CPU 模式加载后约需 **4-5GB** 内存
3. 加上 PostgreSQL 容器（~0.5GB）和 Docker 引擎本身，7.675GB 不够用
4. TEI 的 Candle 后端（CPU）加载 safetensors 模型时，需要将权重全部读入内存

**容器日志特征**：
```
Starting Bert model on Cpu    ← 开始加载
Warming up model              ← 加载完成，开始预热（此时内存峰值）
                              ← OOM Kill，容器退出，restart: unless-stopped 触发重启
Starting Bert model on Cpu    ← 重新开始...
```

**解决方案**（按推荐顺序）：

| 方案 | 说明 | 适用场景 |
|------|------|---------|
| **增加 Docker 内存到 12GB+** | Docker Desktop → Settings → Resources → Memory → 12GB → Apply & Restart | 宿主机 ≥16GB RAM |
| **Python 宿主机跑 Reranker** | 用 FastAPI + FlagEmbedding 在宿主机直接跑，绕过 Docker 内存限制 | 不想改 Docker 配置 |
| **换更小的模型** | bce-reranker-base_v1（279MB，~1GB 内存） | 内存紧张 |
| **用 GPU 模式** | TEI GPU 镜像，VRAM 2.5GB 即可 | 有独立 GPU 且 VRAM 充裕 |

**当前状态**：代码已全部实现并通过编译，Reranker 功能有优雅降级（`app.reranker.enabled=false` 时走原有向量检索），TEI 部署待解决 Docker 内存问题后即可端到端测试

**解决方案**：换用更小的模型 bce-reranker-base_v1（279MB，~1GB 内存），Docker 7.6GB 内存足够运行

**面试知识点**：Docker 容器 OOM 排查
1. `docker inspect <container> --format '{{.State.OOMKilled}}'` — 确认是否 OOM
2. `docker stats` — 实时查看容器内存使用
3. Docker Desktop 默认内存限制可能远小于宿主机实际内存
4. AI 模型推理的内存占用 = 模型权重 + 运行时中间计算（通常为权重的 1.5-2 倍）

### 3.54 TEI CPU 版本返回 JSON 数组 vs GPU 版本返回 JSON 对象

**问题**：`RerankingDocumentPostProcessor` 调用 TEI 的 `/rerank` 端点时报错：
```
Error while extracting response for type [TeiRerankerApi$Response]
```

**原因**：TEI 不同版本/后端返回的 JSON 格式不同：
- **TEI CPU 版本（Candle 后端）**：返回 JSON 数组
  ```json
  [{"index": 0, "score": 0.92}, {"index": 2, "score": 0.45}]
  ```
- **TEI GPU 版本（ORT 后端）**：返回 JSON 对象
  ```json
  {"results": [{"index": 0, "score": 0.92, "text": "..."}]}
  ```

本项目使用的是 TEI CPU 版本，所以返回的是数组格式

**解决**：修改 `RerankingDocumentPostProcessor`，先获取原始 JSON 字符串，判断是 `[` 开头还是 `{` 开头，解析为对应的数据结构

```java
String responseBody = restTemplate.postForObject(rerankUrl, entity, String.class);
String trimmed = responseBody.trim();
if (trimmed.startsWith("[")) {
    // CPU 版本：JSON 数组
    results = objectMapper.readValue(trimmed,
            new TypeReference<List<RerankResult>>() {});
} else {
    // GPU 版本：JSON 对象
    Response response = objectMapper.readValue(trimmed, Response.class);
    results = response.getResults();
}
```

**面试知识点**：TEI (Text Embeddings Inference) 是 HuggingFace 官方推理服务器，不同后端（CPU Candle / GPU ORT）的 API 响应格式可能存在差异，生产环境需做好兼容处理

### 3.55 QueryRewriteService 模型复述指令而非改写问题（ChatClient 消息角色混淆）

**问题**：Query Rewrite 功能中，qwen3:1.7b 模型将 Prompt 模板中的指令文字原样复述出来，而非执行改写任务
- 用户输入：`并发是什么` → 模型输出：`请将用户问题改写为技术问题（如"线程池"→"Java线程池"）`
- 用户输入：`java对象是什么` → 模型输出：`请将用户的问题改写成更清晰的技术问题描述。`

**根因**：`QueryRewriteService` 使用 `chatClient.prompt(prompt).call().content()` 调用方式错误：
```java
// ❌ 错误做法：把整个 Prompt 模板内容（含指令+用户问题）作为一条 UserMessage
PromptTemplate template = new PromptTemplate(templateContent);
String promptText = template.render(Map.of("query", query));  // 拼接成一段文本
Prompt prompt = new Prompt(List.of(new UserMessage(promptText)));
String rewritten = chatClient.prompt(prompt).call().content();
```

`ChatClient` 是高级封装，当整个 Prompt 模板（指令+问题）作为一条 UserMessage 传入时，模型**分不清哪部分是指令、哪部分是要处理的内容**，于是把指令文字原样复述。

**解决**：参考 `QuestionGenerateTool` 的正确做法，改用 `OllamaChatModel` 直接调用 + `SystemMessage/UserMessage` 分离：
```java
// ✅ 正确做法：SystemMessage 放指令，UserMessage 放用户问题，角色分离
Prompt prompt = new Prompt(List.of(
    new SystemMessage(templateContent),  // 系统指令（改写规则）
    new UserMessage(query)               // 用户输入（原始问题）
), options);

String rewritten = chatModel.call(prompt)
    .getResult()
    .getOutput()
    .getText();
```

**三个关键改动**：

| 改动 | 之前 | 之后 | 原因 |
|------|------|------|------|
| **消息角色分离** | 整个内容作为 UserMessage | SystemMessage(指令) + UserMessage(问题) | 模型能区分指令和输入 |
| **调用方式** | `chatClient.prompt(prompt).call().content()` | `chatModel.call(prompt).getResult().getOutput().getText()` | 避免 ChatClient 二次包装 |
| **Prompt 模板** | 含 `<query>` 占位符 + 示例 | 纯指令（query 在 UserMessage 中单独传入） | 减少模型混淆 |

**相关文件**：
- `QueryRewriteService.java` — 核心改写逻辑，已修复
- `query-rewrite.st` — Prompt 模板，已移除 `<query>` 占位符

**面试知识点**：`ChatClient` 是 Spring AI 的高级封装，适合对话场景；`OllamaChatModel` 是底层模型客户端，适合需要精确控制消息结构的场景（如 Tool Calling、Query Rewrite）。消息角色（System/User/Assistant）的正确分离是 LLM 调用的基础。
