# 面试讲故事素材

> 项目练手过程中踩过的坑和做的取舍，面试时主动讲比 feature 本身更值钱

| 素材 | 面试怎么讲 |
|------|-----------|
| TEI Reranker OOM → 换 bce-reranker-base_v1 | "选模型要考虑硬件约束，568M 参数的 bge-reranker 在 6GB VRAM 上 OOM，换 279MB 的 bce-reranker 解决" |
| 查询改写消息角色混淆 | "ChatClient 封装把 SystemMessage 和 UserMessage 混在一起，模型复述指令而不是改写查询，改用 OllamaChatModel 直接调用修复" |
| 混合检索 RRF 融合 | "纯向量检索对专有名词（类名/方法名）效果差，加了 ILIKE 关键词检索 + RRF 加权融合" |
| 流式 Markdown 渲染不完整 | "流式输出时代码块没闭合导致高亮失效，设计了自动补全未关闭标签 + 流式结束后重新完整渲染的策略" |
| PDF 代码块逐行拆散 | "PDF 提取的固有局限，认识到数据质量对 RAG 效果的影响比算法优化更大" |
| DeepSeek thinking 参数注入 | "DeepSeek API 的 thinking 参数是自定义请求体字段，Spring AI 不原生支持。通过 HTTP 拦截器 + ThreadLocal 在请求体中注入，包装 DeepSeekChatModel 管理 ThreadLocal 生命周期，流式场景用 doFinally 清理" |
| 多 Provider 切换架构 | "本地 Ollama 有性能瓶颈，接入 DeepSeek API 云端 LLM。通过 @ConditionalOnProperty + @Primary 实现 ChatModel Bean 按 provider 类型自动装配，ChatOptionsFactory 消除 6 个 Service 对具体 Provider 的硬编码依赖" |
| OpenAI 自动配置条件冲突 | "引入 spring-ai-starter-model-openai 后，@ConditionalOnProperty(matchIfMissing=true) 导致 Ollama 和 OpenAI 两个 Embedding Model 同时创建。需显式设置 spring.ai.model.embedding=ollama 消除歧义" |
| PromptTemplate 模板语法踩坑 | "Spring AI 1.1.0 的 StTemplateRenderer 默认用 {} 分隔符，不是 <> 也不是 {{}}。通过 javap 字节码反编译 Builder 构造函数确认 bipush 123/125（ASCII {/}），而非猜测语法" |
