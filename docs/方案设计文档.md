## 二、Embedding 模型选型分析

### 2.1 为什么选择 BGE-M3

| 对比项 | nomic-embed-text | bge-m3 |
|--------|-----------------|--------|
| 中文支持 | 一般 | **优秀**（智源研究院 BAAI 出品） |
| 语言覆盖 | 主要英文 | **100+ 语言，支持跨语言检索** |
| 向量维度 | 768 | 1024 |
| Ollama 支持 | 有 | **有**（`ollama pull bge-m3`） |
| 检索方式 | Dense | **Dense + Sparse + ColBERT 三种** |
| 分词粒度 | 单一 | **多粒度（Multi-Granularity）** |
| MTEB 中文排名 | 较低 | **前列** |

### 2.2 BGE-M3 核心优势

- **Multi-Functionality**：同时支持 Dense Retrieval、Sparse Retrieval、ColBERT 三种检索
- **Multi-Linguality**：支持 100+ 语言，中文表现优异
- **Multi-Granularity**：支持不同粒度的文本切分，适合 Chunking 场景

### 2.3 后续升级方向

当前使用 Dense Retrieval，后续可升级为混合检索（Dense + Sparse），提升检索精度。

## 六、RAG 流程设计

### 6.1 文档导入流程

```
用户上传文档 (PDF/MD/TXT)
        │
        ▼
① 文件解析 → 纯文本提取
        │
        ▼
② 文本分块 (Chunking)
   - 策略：按 Token 数分段（500 token/段，重叠 50 token）
   - 保留上下文重叠，避免语义断裂
        │
        ▼
③ 文本 → Embedding (bge-m3)
   - 维度：1024
   - 批量调用 Ollama Embedding API
        │
        ▼
④ 向量 + 元数据 → PGVector
   - 元数据包含：domain, doc_id, chunk_index
        │
        ▼
⑤ 更新 knowledge_document 记录
```

### 6.2 RAG 检索增强流程

```
用户提问 "Java线程池核心参数有哪些？"
        │
        ▼
① 问题文本 → Embedding (bge-m3)
        │
        ▼
② 向量相似度检索 PGVector
   - 相似度算法：余弦相似度 (Cosine)
   - 返回 Top-K（默认 K=5）相关文档片段
        │
        ▼
③ 拼接增强 Prompt
   System Prompt（角色设定）
   + 检索到的文档片段（上下文）
   + 用户原始问题
        │
        ▼
④ LLM 生成回答 (qwen3.5:4b)
   - 基于检索到的真实文档内容回答
   - 减少幻觉，提高准确性
        │
        ▼
⑤ 返回增强后的面试题与知识点
```

### 6.3 Chunking 策略

| 参数 | 值 | 说明 |
|------|-----|------|
| chunk_size | 500 tokens | 每段最大 Token 数 |
| chunk_overlap | 50 tokens | 相邻段重叠 Token 数 |
| separator | \n\n | 优先按段落分割 |

Spring AI 提供 `TokenTextSplitter`，可直接使用。

## 八、Tool Calling 设计

### 8.1 工具清单

| 工具 | 描述 | 触发场景 | Phase |
|------|------|----------|-------|
| KnowledgeSearchTool | 从知识库检索面试题和知识点 | 用户问某领域面试内容 | Phase 1 |
| QuestionGenerateTool | 按难度生成面试题 | 用户指定难度和领域 | Phase 1 |
| ProgressTrackerTool | 查询学习进度 | 用户问学习历史 | Phase 2 |

### 8.2 Tool Calling vs 硬编码

| 对比 | 硬编码 | Tool Calling |
|------|--------|-------------|
| 决策者 | 开发者写 if-else | LLM 自主决定是否调用 |
| 灵活性 | 需改代码 | 加一个 Tool 即可 |
| 扩展性 | 差 | 插拔式，开闭原则 |
| 智能 | 无 | LLM 理解上下文后选择合适工具 |

### 8.3 实现方式

```java
@Component
public class KnowledgeSearchTool implements ToolCallback {
    
    @Tool(description = "从面试知识库中检索与用户问题相关的面试题和知识点")
    public String searchKnowledge(
        @ToolParam(description = "搜索关键词") String query,
        @ToolParam(description = "技术领域：java/python/ai") String domain,
        @ToolParam(description = "返回结果数量，默认5") int topK
    ) {
        // Embedding query → PGVector 相似度检索 → 返回文档片段
    }
}

// QuestionGenerateTool 使用 @Lazy + OllamaChatModel 避免循环依赖
@Component
public class QuestionGenerateTool {

    private final OllamaChatModel chatModel;

    @Lazy
    public QuestionGenerateTool(OllamaChatModel chatModel) {
        this.chatModel = chatModel;
    }

    @Tool(description = "根据技术领域和难度生成面试题")
    public String generateQuestions(
        @ToolParam(description = "技术领域") String domain,
        @ToolParam(description = "难度级别") String difficulty,
        @ToolParam(description = "生成数量") int count
    ) {
        // 使用 OllamaChatModel 直接调用，避免循环依赖
    }
}
```

> **注意**：Tool 不在 SpringAiConfig 的 ChatClient Bean 中注册，而是在 Service 层通过 `chatClient.prompt().tools(...)` 动态注册，避免循环依赖。

## 十一、面试亮点总结

1. **BGE-M3 选型**：中文 Embedding 主流，支持 Dense + Sparse + ColBERT 三种检索
2. **Tool Calling**：LLM 自主决策工具调用，体现 Agent 思想，替代硬编码 if-else
3. **Prompt 外部化管理**：.st 模板文件解耦 Prompt 与代码
4. **RAG 全链路**：文档上传 → Chunking → Embedding → 向量检索 → 增强生成
5. **PGVector 一库两用**：关系数据 + 向量数据统一管理
6. **Spring AI 1.0 GA**：使用最新稳定版，适配未来生态
