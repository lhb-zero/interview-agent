### 5.1 RAG 进阶优化路线图（2026-04-25 规划）

> 当前架构：入门级 RAG（文档 → 清洗 → 分块 → Embedding → 向量检索 → Prompt拼接 → LLM生成）
> 
> 目标架构：生产级 RAG（用户问题 → 查询改写 → 混合检索 → Re-ranking → 上下文压缩 → LLM生成）

#### Phase 1：Re-ranking 重排序 🔶 代码完成，部署待解决

| 项目 | 说明 |
|------|------|
| **方案** | HuggingFace TEI Docker + bge-reranker-v2-m3 Cross-Encoder 精排 |
| **原因** | Ollama 不支持 /v1/rerank 端点（截至 2026-04 仍未实现），TEI 是 HuggingFace 官方推理服务器，原生支持 /rerank；用 1.7b 小模型做 LLM Rerank 不准确 |
| **原理** | 向量检索 Top-20（召回阶段）→ TEI Cross-Encoder 精排 Top-5（精排阶段）→ 拼入 Prompt |
| **实现** | 1. `RerankerProperties` — @ConfigurationProperties 映射 app.reranker.* 配置 |
| | 2. `TeiRerankerApi` — TEI /rerank 端点请求/响应数据结构 |
| | 3. `RerankingDocumentPostProcessor` — 核心精排逻辑（HTTP 调用 + 排序 + 降级） |
| | 4. `RagServiceImpl.searchSimilarDocs()` — 扩大召回量 + 调用 Reranker |
| | 5. `KnowledgeSearchTool.searchKnowledge()` — Tool Calling 模式同样集成 Reranker |
| **降级策略** | Reranker 不可用时自动降级为原始向量检索结果，不影响核心对话功能 |
| **配置** | `app.reranker.enabled=true`, `url=http://localhost:8081`, `candidate-count=20`, `top-k=5` |
| **延迟** | CPU 模式 20 个文档精排约 100-300ms |

#### Phase 2：混合检索 Hybrid Search（召回率提升最大）

| 项目 | 说明 |
|------|------|
| **方案** | PGVector 向量检索 + PostgreSQL tsvector 全文检索 → RRF 融合排序（方案A） |
| **原因** | 不引入新组件（ES/Chroma），PostgreSQL 一站式，架构简洁 |
| **原理** | 向量检索擅长语义匹配（"什么是线程池"），关键词检索擅长精确匹配（"ReentrantLock"） |
| **实现要点** | 1. 导入时同步生成 `tsvector` 并存入 PG 全文索引列 |
| | 2. 检索时双路查询：PGVector similaritySearch + tsvector to_tsquery |
| | 3. RRF（Reciprocal Rank Fusion）融合两路结果 |
| **中文分词** | PostgreSQL 默认分词对中文不友好，需安装 `pg_jieba` 或 `zhparser` 扩展，或用 jieba Java 库预分词后存入 |
| **预期效果** | 专有名词（类名/方法名）检索命中率大幅提升 |

#### Phase 3：查询改写 Query Rewriting（解决用户提问质量差的问题）

| 项目 | 说明 |
|------|------|
| **方案** | Spring AI 内置 `RewriteQueryTransformer` |
| **原因** | Spring AI 1.1.0 已封装好，配置即用 |
| **原理** | 用户原始问题 → LLM 改写为更适合检索的形式 → 用改写后的 query 检索 |
| **示例** | "怎么解决死锁？" → 改写为 "Java 死锁的避免与检测方法" |
| **实现要点** | 1. 从 `QuestionAnswerAdvisor` 迁移到 `RetrievalAugmentationAdvisor` |
| | 2. 配置 `RewriteQueryTransformer` 作为 queryTransformer |
| **预期效果** | 口语化/模糊提问的检索效果提升 |

#### Phase 4：迁移到 RetrievalAugmentationAdvisor 架构

| 项目 | 说明 |
|------|------|
| **当前** | `RagServiceImpl` 手动调用 VectorStore + 手动拼 Prompt |
| **目标** | 使用 Spring AI 1.1.0 的 `RetrievalAugmentationAdvisor` 模块化架构 |
| **好处** | 查询改写/检索/后处理全部模块化，方便组合不同 RAG 策略 |
| **架构对比** | 当前：`RagServiceImpl.search() → VectorStore.similaritySearch() → 手动拼Prompt → ChatClient.call()` |
| | 目标：`ChatClient.prompt().advisors(RetrievalAugmentationAdvisor).advisors(自定义PostProcessor).user(question).call()` |

#### Phase 5：上下文压缩 + RAG 评估

| 项目 | 说明 |
|------|------|
| **上下文压缩** | 检索到的文档片段可能很长，用 LLM 提取与问题相关的部分，减少 Token 浪费 |
| **RAG 评估** | 使用 RAGAS 框架量化：Context Precision / Context Recall / Faithfulness / Answer Relevancy |
